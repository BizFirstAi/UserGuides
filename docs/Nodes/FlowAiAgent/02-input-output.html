Input & Output

Output ports, data schemas, tool call structure, and expression paths for consuming agent results in downstream nodes.

Output Ports

PortTriggered WhenData Available
success The agent produced a final answer before hitting max_turns. The model declared the goal complete. Full output object: result, reasoning, tool_calls, turns_used, model_used
error The agent hit max_turns without completing, a tool invocation failed fatally, or the model API returned an error. error_message, error_code, turns_used, partial_reasoning (reasoning trace up to the point of failure)

Output Data Schema — success port

FieldTypeDescription
resultstring / objectThe agent's final answer. May be a plain text string or structured JSON depending on how the agent was instructed to respond via system_prompt.
reasoningstringFull chain-of-thought text produced by the agent across all turns. Useful for debugging and audit. Not returned to end users unless deliberately included.
tool_callsarrayOrdered list of every tool the agent invoked. Each element contains tool_name, inputs, output, and turn_number.
turns_usedintegerHow many reasoning turns were executed. Compare against max_turns to detect near-limit runs.
model_usedstringThe model ID that was actually used (useful when using model aliases or fallback logic).
tokens_inputintegerTotal input tokens consumed across all turns.
tokens_outputintegerTotal output tokens generated across all turns.

tool_calls Array — Item Schema

FieldTypeDescription
turn_numberintegerWhich reasoning turn this tool call occurred in (1-indexed).
tool_namestringName of the tool that was invoked.
inputsobjectThe input parameters the agent passed to the tool. Matches the tool's registered input schema.
outputanyThe result returned by the tool to the agent.
successbooleantrue if the tool completed without error, false if the tool returned an error (the agent may recover and try differently).
duration_msintegerWall-clock time the tool took to execute in milliseconds.

Expression Paths for Downstream Nodes

After the FlowAiAgent node, use the following BizFirst expressions to access output data in downstream node configurations:

ExpressionReturns
{@ $node.agentNode.result }The agent's final answer (string or object)
{@ $node.agentNode.reasoning }Full reasoning trace text
{@ $node.agentNode.turns_used }Number of turns used
{@ $node.agentNode.tool_calls }Full tool call array
{@ $node.agentNode.tool_calls[0].tool_name }Name of the first tool called
{@ $node.agentNode.tool_calls[0].output }Output of the first tool call
{@ $node.agentNode.model_used }Model that executed the agent
{@ $node.agentNode.tokens_input }Total input tokens
Structured result output: For predictable downstream processing, instruct the agent to respond in JSON via the system_prompt: "Your final answer must be a valid JSON object matching this schema: …". Then access individual fields with expressions like {@ $node.agentNode.result.customerSentiment }.

Error port handling: Always connect a handler to the error port. At minimum, log the error_message and partial_reasoning to your audit system and trigger an alert. For customer-facing workflows, route the error port to a graceful fallback response rather than surfacing the raw error to the end user.

Using result as a Structured JSON Object

When the agent's system_prompt instructs it to respond in JSON, the result field is automatically parsed into a workflow object. You can then access individual fields directly:

// Agent system_prompt: "Your final response must be valid JSON with fields: category, priority, queue_id"

// Access structured fields downstream:
{@ $node.agentNode.result.category }
{@ $node.agentNode.result.priority }
{@ $node.agentNode.result.queue_id }

If the agent does not produce valid JSON (e.g. it wraps the JSON in markdown code fences), the result field is returned as a plain string. Add explicit instructions like "Respond with ONLY a JSON object — no markdown, no explanation, just the raw JSON" to ensure reliable parsing.

Inspecting tool_calls for Debugging

The tool_calls array is the primary debugging surface for agent workflows. When an agent produces unexpected results, examine:

  • tool_name sequence — did the agent call the right tools in the expected order?
  • inputs per tool — did the agent construct the tool parameters correctly from the available context?
  • output per tool — did the tool return the expected data? If output is empty or wrong, the issue is likely in the tool's implementation or data, not the agent.
  • success flags — were any tools marked success: false? The agent may have attempted recovery, or it may have proceeded with incomplete data.
  • turns_used — if turns_used == max_turns, the agent ran out of turns. Increase max_turns or simplify the goal.

token Usage Monitoring

FieldWhat to Watch For
tokens_inputGrows with each turn as context accumulates. High values (>20,000) indicate the agent is using many turns or has very long tool outputs. Consider summarising tool outputs in the tool definition.
tokens_outputReflects the volume of reasoning text and final answer generated. Very high values may mean the agent is over-explaining — tighten the system_prompt to be more concise.
turns_usedIf consistently close to max_turns, either increase the limit or decompose the goal into simpler sub-tasks using multiple agent nodes in sequence.
Chaining agents: For very complex goals that require more than 15–20 tool calls, consider decomposing the goal into two sequential FlowAiAgent nodes. The first agent collects and prepares data; the second agent analyses and produces the final output. Pass the first agent's result as the context for the second agent.
