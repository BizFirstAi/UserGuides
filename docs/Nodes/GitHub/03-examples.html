Examples

Five GitHub node configurations for bug reports, PRs, deployment comments, sprint tracking, and issue queries.

Example 1: Auto-Create Bug Issue from Monitoring Alert

Scenario: A monitoring workflow detects a production error. Create a GitHub issue with full context automatically.

{
  "credential_id": "github-pat-devops",
  "operation": "createIssue",
  "owner": "{{ env.GITHUB_ORG }}",
  "repo": "{{ vars.affected_service_repo }}",
  "title": "[{{ vars.severity }}] {{ vars.error_type }} — {{ vars.environment }}",
  "body": "## Incident Summary\n\n**Service:** {{ vars.service_name }}\n**Environment:** {{ vars.environment }}\n**Severity:** {{ vars.severity }}\n**Detected At:** {{ vars.detected_at }}\n\n## Error Details\n\n```\n{{ vars.error_message }}\n```\n\n## Stack Trace\n\n```\n{{ vars.stack_trace }}\n```\n\n## Monitoring Dashboard\n{{ vars.dashboard_url }}\n\n*Auto-created by BizFirst workflow — Incident ID: {{ workflow.id }}*",
  "labels": ["bug", "{{ vars.severity_label }}", "{{ vars.environment_label }}"],
  "assignees": ["{{ vars.oncall_github_username }}"]
}
Expected outcome: A fully detailed GitHub issue is created with error details, stack trace, and a monitoring link. The on-call engineer is assigned and notified by GitHub. Issue URL is passed to a Slack notification node.

Example 2: Create Pull Request After Auto-Configuration Update

Scenario: A workflow has auto-generated a configuration file update on a feature branch. Open a PR to main for review.

{
  "credential_id": "github-pat-devops",
  "operation": "createPullRequest",
  "owner": "{{ env.GITHUB_ORG }}",
  "repo": "infra-config",
  "title": "Auto: Update {{ vars.config_key }} for {{ vars.environment }} — {{ vars.change_id }}",
  "head": "auto/config-update-{{ vars.change_id }}",
  "base": "main",
  "body": "## Automated Configuration Update\n\n**Change ID:** {{ vars.change_id }}\n**Config Key:** {{ vars.config_key }}\n**New Value:** {{ vars.new_value }}\n**Changed By:** BizFirst Workflow ({{ workflow.id }})\n**Approved By:** {{ vars.approver_name }} on {{ vars.approval_time }}\n\n## Test Results\n- Configuration validation: PASSED\n- Dry-run deployment: PASSED\n\nCloses #{{ vars.related_issue_number }}",
  "reviewers": ["{{ vars.infra_team_lead }}"],
  "draft": false
}
Expected outcome: A detailed PR is opened, the infra team lead is requested as reviewer, and the related issue is linked for auto-close on merge.

Example 3: Post Deployment Status Comment on PR

Scenario: After deploying to staging, comment on the PR with deployment details and the staging URL.

{
  "credential_id": "github-pat-devops",
  "operation": "commentOnIssue",
  "owner": "{{ env.GITHUB_ORG }}",
  "repo": "{{ vars.repo_name }}",
  "issue_number": "{{ vars.pr_number }}",
  "body": "## Staging Deployment — {{ vars.deploy_status_emoji }} {{ vars.deploy_status }}\n\n| Property | Value |\n|---|---|\n| **Environment** | Staging |\n| **Version** | `{{ vars.version }}` |\n| **Deployed At** | {{ vars.deployed_at }} |\n| **Duration** | {{ vars.deploy_duration_seconds }}s |\n| **Staging URL** | {{ vars.staging_url }} |\n\n### Test Results\n- Unit tests: {{ vars.unit_test_status }}\n- Integration tests: {{ vars.integration_test_status }}\n\n*Deployed by BizFirst CI/CD workflow*"
}
Expected outcome: A formatted table comment is posted on the PR with deployment details and a link to the staging environment for reviewers to test.

Example 4: List Open High-Priority Issues for Sprint Planning

Scenario: Every Monday, fetch all open high-priority issues to generate a sprint planning summary email.

{
  "credential_id": "github-pat-devops",
  "operation": "listIssues",
  "owner": "{{ env.GITHUB_ORG }}",
  "repo": "main-application",
  "state": "open",
  "labels": "high-priority,bug",
  "sort": "created",
  "per_page": 50
}
Expected outcome: Returns all open high-priority bugs. The issues array is passed to a Transform node to build a formatted HTML table, then sent via the Email (SMTP) node to the engineering manager for sprint prioritisation.

Example 5: Merge Pull Request After Approval

Scenario: After an automated test workflow confirms all checks pass and a human approver clicks "Merge" in a Slack sendAndWait message, merge the PR.

{
  "credential_id": "github-pat-devops",
  "operation": "mergeAsync",
  "owner": "{{ env.GITHUB_ORG }}",
  "repo": "{{ vars.repo_name }}",
  "pull_number": "{{ vars.pr_number }}",
  "merge_method": "squash",
  "commit_message": "{{ vars.pr_title }} (#{{ vars.pr_number }})\n\nApproved by: {{ vars.approver_name }}\nApproval time: {{ vars.approval_time }}\nWorkflow: {{ workflow.id }}"
}
Expected outcome: PR is squash-merged with a descriptive commit message including approver details for full audit trail. The merge commit SHA is stored in output for deployment workflow triggering.
