Examples

Five Slack node examples covering deployment notifications, interactive approvals, incident alerts, file uploads, and personalised DMs.

Example 1: Deployment Notification with Block Kit

Scenario: Post a colour-coded deployment result to the #deployments channel after a CI/CD pipeline completes.

{
  "credential_id": "slack-bot-token",
  "resource": "message",
  "operation": "send",
  "channel": "#deployments",
  "text": "Deployment {{ vars.deploy_status }}: {{ vars.service_name }} v{{ vars.version }}",
  "blocks": [
    {
      "type": "header",
      "text": { "type": "plain_text", "text": "{{ vars.deploy_status_emoji }} Deployment {{ vars.deploy_status }}" }
    },
    {
      "type": "section",
      "fields": [
        { "type": "mrkdwn", "text": "*Service:*\n{{ vars.service_name }}" },
        { "type": "mrkdwn", "text": "*Version:*\nv{{ vars.version }}" },
        { "type": "mrkdwn", "text": "*Environment:*\n{{ vars.environment }}" },
        { "type": "mrkdwn", "text": "*Duration:*\n{{ vars.deploy_duration_seconds }}s" }
      ]
    },
    {
      "type": "actions",
      "elements": [
        {
          "type": "button",
          "text": { "type": "plain_text", "text": "View Logs" },
          "url": "{{ vars.logs_url }}"
        }
      ]
    }
  ]
}
Expected outcome: Rich Block Kit message with service details and a direct link to deployment logs posted in #deployments.

Example 2: Interactive Purchase Approval (sendAndWait)

Scenario: Send an approval request to a manager's DM. Suspend the workflow until they click Approve or Reject.

{
  "credential_id": "slack-bot-token",
  "resource": "message",
  "operation": "sendAndWait",
  "channel": "{{ vars.manager_slack_id }}",
  "text": "Purchase approval required: #{{ vars.pr_number }}",
  "wait_for_response": true,
  "resume_actions": ["approve_purchase", "reject_purchase"],
  "timeout_hours": 48,
  "blocks": [
    {
      "type": "section",
      "text": { "type": "mrkdwn", "text": "*Purchase Request #{{ vars.pr_number }}* requires your approval.\n*Amount:* {{ vars.pr_amount_formatted }}\n*Vendor:* {{ vars.vendor_name }}\n*Justification:* {{ vars.justification }}" }
    },
    {
      "type": "actions",
      "elements": [
        { "type": "button", "text": { "type": "plain_text", "text": "Approve" }, "style": "primary", "action_id": "approve_purchase" },
        { "type": "button", "text": { "type": "plain_text", "text": "Reject" }, "style": "danger", "action_id": "reject_purchase" }
      ]
    }
  ]
}
Expected outcome: Workflow suspends. Manager sees the request in Slack DM. On button click, workflow resumes on the response port with action_id indicating the decision.

Example 3: IT Incident Alert with Urgency

Scenario: Broadcast a P1 incident to the #incidents channel and mention the on-call engineer by their Slack user ID.

{
  "credential_id": "slack-bot-token",
  "resource": "message",
  "operation": "send",
  "channel": "#incidents",
  "text": "P1 INCIDENT: {{ vars.service_name }} is DOWN. On-call: <@{{ vars.oncall_slack_id }}>",
  "blocks": [
    {
      "type": "header",
      "text": { "type": "plain_text", "text": ":red_circle: P1 INCIDENT — Immediate Action Required" }
    },
    {
      "type": "section",
      "fields": [
        { "type": "mrkdwn", "text": "*Service:*\n{{ vars.service_name }}" },
        { "type": "mrkdwn", "text": "*Severity:*\nP1 — Critical" },
        { "type": "mrkdwn", "text": "*Started:*\n{{ vars.incident_start_time }}" },
        { "type": "mrkdwn", "text": "*On-Call:*\n<@{{ vars.oncall_slack_id }}>" }
      ]
    },
    {
      "type": "section",
      "text": { "type": "mrkdwn", "text": "*Error:* {{ vars.error_message }}\n*Dashboard:* {{ vars.dashboard_url }}" }
    }
  ]
}
Expected outcome: Loud, formatted incident alert posted to #incidents with the on-call engineer directly mentioned for immediate notification.

Example 4: Upload a Weekly Report PDF

Scenario: After generating a weekly business report PDF, upload it to the #management channel with a summary message.

{
  "credential_id": "slack-bot-token",
  "resource": "file",
  "operation": "upload",
  "channels": ["#management", "#finance"],
  "filename": "Weekly_Report_{{ vars.week_ending_date }}.pdf",
  "file_data": "{{ vars.report_pdf_base64 }}",
  "title": "Weekly Business Report — Week Ending {{ vars.week_ending_date }}",
  "initial_comment": "Here is the weekly business report. Key highlights: Revenue {{ vars.weekly_revenue }} | New Customers {{ vars.new_customers }} | MRR {{ vars.mrr_formatted }}"
}
Expected outcome: PDF uploaded to both #management and #finance channels with an inline summary comment. Stakeholders can view or download directly from Slack.

Example 5: Personalised Onboarding DM to New Employee

Scenario: When an HR workflow creates a new user account, send a personalised welcome DM with onboarding resources.

{
  "credential_id": "slack-bot-token",
  "resource": "message",
  "operation": "send",
  "channel": "{{ vars.new_employee_slack_id }}",
  "blocks": [
    {
      "type": "header",
      "text": { "type": "plain_text", "text": "Welcome to the team, {{ vars.employee_first_name }}! :wave:" }
    },
    {
      "type": "section",
      "text": { "type": "mrkdwn", "text": "We are thrilled to have you joining *{{ vars.department }}*. Here are a few things to get you started:" }
    },
    {
      "type": "section",
      "text": { "type": "mrkdwn", "text": ":bookmark: *<{{ vars.handbook_url }}|Employee Handbook>*\n:computer: *<{{ vars.it_setup_url }}|IT Setup Guide>*\n:calendar: *<{{ vars.onboarding_calendar_url }}|Onboarding Calendar>*" }
    },
    {
      "type": "context",
      "elements": [{ "type": "mrkdwn", "text": "Your manager is <@{{ vars.manager_slack_id }}>. Don't hesitate to reach out!" }]
    }
  ]
}
Expected outcome: New employee receives a warm, formatted welcome DM with direct links to key onboarding resources and their manager's mention.
