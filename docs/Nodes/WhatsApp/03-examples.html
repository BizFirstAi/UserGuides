Examples

Five WhatsApp node configurations covering OTP, order updates, templates, media, and delivery tracking.

Example 1: OTP Delivery via Text Message

Scenario: Send a one-time password to a customer's WhatsApp during a login or transaction verification flow.

{
  "credential_id": "whatsapp-business-api",
  "operation": "send",
  "phone_number": "{{ vars.customer_phone }}",
  "message_type": "text",
  "message_text": "Your {{ env.COMPANY_NAME }} verification code is: *{{ vars.otp_code }}*\n\nThis code expires in 10 minutes. Do not share it with anyone."
}
Expected outcome: OTP delivered to customer's WhatsApp. Bold formatting applied to the code for readability. The 10-minute expiry is enforced by an upstream Delay + VariableAssignment node that invalidates the token.

Example 2: Order Shipped Template Notification

Scenario: When an order ships, send a pre-approved WhatsApp template message with the tracking details. This is sent outside a customer service window, so a template is required.

{
  "credential_id": "whatsapp-business-api",
  "operation": "sendTemplate",
  "phone_number": "{{ vars.customer_phone }}",
  "template_name": "order_shipped_notification",
  "template_language": "en_US",
  "parameters": [
    "{{ vars.customer_first_name }}",
    "{{ vars.order_number }}",
    "{{ vars.carrier_name }}",
    "{{ vars.tracking_number }}",
    "{{ vars.estimated_delivery_date }}",
    "{{ vars.tracking_url }}"
  ]
}
Expected outcome: Customer receives a branded shipping notification: "Hi Jane, your order #ORD-2025-8471 has shipped via FedEx (tracking: 794644792798). Estimated delivery: 17 Nov. Track: https://..."

Example 3: Document Invoice via Media Message

Scenario: Within a 24-hour support window, send the customer their invoice as a PDF document attachment.

{
  "credential_id": "whatsapp-business-api",
  "operation": "send",
  "phone_number": "{{ vars.customer_phone }}",
  "message_type": "document",
  "media_url": "{{ vars.invoice_pdf_url }}",
  "media_filename": "Invoice_{{ vars.invoice_number }}.pdf",
  "media_caption": "Invoice #{{ vars.invoice_number }} for {{ vars.invoice_amount_formatted }}. Due: {{ vars.invoice_due_date }}."
}
Expected outcome: Customer receives the PDF invoice directly in WhatsApp with a clear filename and caption. The media_url must be publicly accessible or a pre-signed S3 URL.

Example 4: Appointment Reminder Template

Scenario: 24 hours before a scheduled appointment, send a reminder via approved template with confirmation and rescheduling options.

{
  "credential_id": "whatsapp-business-api",
  "operation": "sendTemplate",
  "phone_number": "{{ vars.patient_phone }}",
  "template_name": "appointment_reminder_v3",
  "template_language": "en_US",
  "parameters": [
    "{{ vars.patient_first_name }}",
    "{{ vars.appointment_date }}",
    "{{ vars.appointment_time }}",
    "{{ vars.doctor_name }}",
    "{{ vars.clinic_address }}",
    "{{ vars.reschedule_url }}"
  ]
}
Expected outcome: Patient receives an appointment reminder with all details. Reply handling via a webhook trigger workflow can capture confirmation or rescheduling requests.

Example 5: Check Delivery Status After Send

Scenario: After sending a critical compliance notification, poll its delivery status to confirm receipt before proceeding with the workflow.

{
  "credential_id": "whatsapp-business-api",
  "operation": "getStatus",
  "message_id": "{{ nodes.sendComplianceNotice.output.message_id }}"
}
This node is typically placed after a short Delay (e.g. 30 seconds) to allow delivery. Use a Switch node downstream on {{ nodes.checkStatus.output.status }} to route: delivered → continue, failed → escalate via alternate channel.
