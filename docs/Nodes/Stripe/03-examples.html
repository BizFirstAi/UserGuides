Examples

Four examples: create payment intent, customer onboarding to subscription, refund approval workflow, and invoice-to-cash automation.

Example 1 — Create Payment Intent

Create a PaymentIntent for a $49.00 USD purchase tied to an existing customer, with order metadata for reconciliation.

PropertyValue
operationcreate_payment_intent
credential_idstripe-live
amount4900
currencyusd
customer$ctx.stripe_customer_id
descriptionOrder #$ctx.order_id
metadata{"order_id": "$ctx.order_id", "product": "$ctx.product_name"}
idempotency_key$ctx.order_id
The client_secret in the output is returned to your frontend. Stripe.js uses it to present the payment UI to the customer without any additional server interaction.

Example 3 — Refund via Agent Approval

A customer requests a refund. Route to a CS agent for approval, then issue the Stripe refund automatically on approval.

  1. WebhookTrigger — receives refund request: { payment_intent_id, reason, amount }
  2. Approval (HIL) — send to CS team: "Customer requests refund of $49.00 for order #ORD-421. Reason: product defect. Approve or reject?"
  3. Stripe (create_refund) — on approval:
    payment_intent: $ctx.payment_intent_id
    amount: $ctx.amount
    reason: requested_by_customer
  4. EmailSmtp — send refund confirmation to customer
  5. MongoDB (update) — log refund event in order record
The HIL approval step prevents unauthorized refunds while keeping the automated path fast — the agent approves in Slack or email and the refund is processed immediately.

Example 4 — Invoice-to-Cash Automation

Monthly billing run: create and send a Stripe invoice to each enterprise customer, then track payment via webhook.

  1. ScheduledTrigger — runs on the 1st of each month at 08:00
  2. MongoDB (find) — retrieve all customers with billing_type: "invoice"
  3. Loop — iterate over customer list
  4. Stripe (create_invoice) — per customer:
    customer: $loop.stripe_customer_id
    collection_method: send_invoice
    days_until_due: 30
    auto_advance: true
  5. MongoDB (update) — store invoice_id and hosted_invoice_url in customer record
  6. EmailSmtp — send invoice notification with hosted_invoice_url link
Payment confirmation: Invoice payment confirmation arrives via Stripe webhook (invoice.payment_succeeded). Set up a separate WebhookTrigger workflow to handle this event and update your CRM when invoices are paid.

Example 5 — Cancel Subscription at Period End

When a customer requests cancellation through your app, cancel their Stripe subscription at the end of the current billing period, send a cancellation confirmation, and trigger an offboarding sequence.

  1. WebhookTrigger — receives cancellation request from app: { user_id, subscription_id, reason }
  2. Stripe (cancel_subscription) — cancel gracefully at period end:
    subscription_id: $ctx.subscription_id
    cancel_at_period_end: true
  3. IfCondition — check cancel_at_period_end == true
  4. EmailSmtp — send cancellation confirmation with access end date:
    "Your subscription is cancelled. You'll retain access until [period end date]."
  5. MongoDB (update) — record cancellation_date and access_end_date in user record
  6. CRM (update) — mark customer as churned with cancellation reason

The cancel_at field in the Stripe output gives the Unix timestamp of when the subscription will definitively end — convert this to a human-readable date for the customer email using a DataMapping node.

// DataMapping to format the access end date
{
  "access_end": "$datetime.format($ctx.cancel_at, 'MMMM Do, YYYY')"
}

Example 6 — Payment Failure Recovery

When Stripe reports a subscription as past_due via webhook, trigger an automated recovery sequence to update the payment method before suspension.

  1. WebhookTrigger — invoice.payment_failed event from Stripe
  2. Stripe (retrieve_payment_intent) — get the failed PaymentIntent details and decline reason
  3. IfCondition — branch on decline_code:
    insufficient_funds → send "retry in 3 days" email
    card_expired → send "update payment method" email with portal link
    do_not_honor → escalate to billing team
  4. EmailSmtp — personalised recovery email with Stripe Customer Portal link
  5. Delay — wait 3 days
  6. Stripe (retrieve_payment_intent) — check if status is now succeeded
  7. IfCondition — if still failed after 7 days: suspend account and notify
Stripe automatically retries failed payments on its Smart Retries schedule. This workflow adds a customer communication layer on top of Stripe's automated retries — keeping customers informed and reducing involuntary churn.
