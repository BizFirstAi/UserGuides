Step 1: Defining Errors

Create and structure error codes for your application.

What it covers: Error code naming convention, error categories, and how to assign unique error codes in your application code.

Error Code Naming Convention

All error codes follow a standardized format for consistency and searchability:

Error-{PRODUCT}-{NUMBER}

Examples:
Error-FlowStudio-100090
Error-AppStudio-200001
Error-ProcessEngine-150025

Components Explained

Component Format Example Description
Prefix Error- Error- Fixed prefix for all error codes
Product PascalCase (no spaces) FlowStudio The application name (as provided by developers)
Number 5-6 digit unique ID 100090 Unique numeric identifier within that product

Assigning Error Numbers

Error number ranges should be organized by product and feature area:

Number Ranges by Feature

FlowStudio:
  100000-100999 → Workflow execution errors
  101000-101999 → Node validation errors
  102000-102999 → Data binding errors
  103000-103999 → Permission/security errors

AppStudio:
  200000-200999 → Widget system errors
  201000-201999 → Layout errors
  202000-202999 → Data binding errors
  203000-203999 → Permission/security errors

Developers are responsible for choosing unique numbers within their product's namespace.

Creating Error in Code

When throwing errors in your application, include the error code in your exception:

C# Example

throw new ApplicationException(
  "Error-FlowStudio-100090: Cannot execute node with invalid input"
);

JavaScript/TypeScript Example

throw new Error(
  "Error-FlowStudio-100090: Cannot execute node with invalid input"
);

Best Practices

  • Always include the error code at the start of the message
  • Make the message descriptive and actionable
  • Include context (parameter names, values, constraints)
  • Never include sensitive data (passwords, API keys, tokens)

Error Categories

When defining errors, classify them by category:

Category Typical Range Description
Validation *-000-*001999 Input validation, constraint violations, invalid configuration
Execution *-000-*000999 Runtime errors, missing data, failed operations
Permission *-003-*003999 Authorization failures, insufficient privileges
System *-999-*999999 Integration failures, external service errors

Picking a New Error Code (Non-Existing)

When you need to create a new error code that doesn't exist yet, follow this process:

Step 1: Determine Your Product

First, identify which product owns this error:

  • FlowStudio — Workflow editor, canvas, node system
  • AppStudio — No-code app builder, widgets, layout
  • AtlasForms — Form builder, controls, validation
  • Octopus — AI agents, memory, RAG
  • Passport — Identity, SSO, RBAC
  • WorkDesk — Task management, HIL inbox
  • BizFirstObserve — Observability, logs, metrics
  • EdgeStream — Real-time messaging, SignalR
  • Other — Custom, integration, or support product

Step 2: Identify Feature Area

Choose the feature area within your product:

Range Typical Category
*-0000-*0999 Core/General errors
*-0100-*0999 Execution/Runtime
*-1000-*1999 Validation/Input
*-2000-*2999 Data/Binding
*-3000-*3999 Permission/Security
*-9000-*9999 Integration/External

Step 3: Find Next Available Number

Method A: Check ErrorCodes Index

  1. Open C:\BizFirstGO_FI_AI\UserGuides\docs\WebSites\ErrorManagement\ErrorCodes\
  2. List all existing files: Error-{PRODUCT}-*.html
  3. Find the highest number in your product's range
  4. Add 1 to get your new error code

Example for FlowStudio:

Existing codes found:
  Error-FlowStudio-100090.html

Next available: 100091

Your new code: Error-FlowStudio-100091

Method B: Search in Code

Search your product's codebase for error code references:

# Search for all error codes in your codebase
grep -r "Error-FlowStudio-" src/
grep -r "Error-FlowStudio-" packages/

# Find the highest number
grep -r "Error-FlowStudio-" src/ | grep -o "[0-9]\{6\}" | sort -n | tail -1

Step 4: Validate Uniqueness

Before committing to a number, verify:

# Check if this exact code already exists in documentation
ls ErrorManagement/ErrorCodes/Error-FlowStudio-100091.html

# Check if it's referenced in code
grep -r "Error-FlowStudio-100091" .

# Should return no results

Step 5: Reserve and Document

Once you've picked your number:

  1. Add it to your code with the error code
  2. Create documentation page: Error-FlowStudio-100091.html
  3. Update ErrorCodes/Index.html to list the new error
  4. Commit both the code change and documentation together

Reserving Error Codes

To avoid conflicts:

  1. Check the ErrorCodes/ directory for existing codes in your range
  2. Follow the "Picking a New Error Code" process above
  3. Document the range in your service's README or developer guide
  4. Commit your error code documentation at the same time as your code change

Common Mistakes

❌ Avoid:
  • Using the same error code for different errors
  • Hardcoding error codes without documentation
  • Inconsistent code formatting (Error_FlowStudio instead of Error-FlowStudio)
  • Creating error codes that duplicate existing ones
✓ Do:
  • Choose unique, sequential numbers within your product
  • Document the error immediately after creating the code
  • Use consistent naming across all products
  • Test that the error code appears in your application's output
